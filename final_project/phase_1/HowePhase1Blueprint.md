## 1. Overview

Phase 1 establishes the foundation of the Mini Registrar System. The focus is admin authentication: reading a manually created admin CSV file and implementing a secure login control loop that locks out an admin after five failed attempts.

This phase also introduces the full OOP class structure (Extra Credit Option 2). All three classes — `User`, `Admin(User)`, and `Student(User)` — are defined in `HoweClasses.py` from day one. `Admin` and `Student` methods that belong to later phases are included as stubs. Building OOP from the start avoids retrofitting and makes the project a stronger portfolio artifact.

No student or course data is handled in this phase. The admin menu introduced here will serve as the entry point for all future phases.

## 5. OOP Class Design

### 5.1 `User` Base Class (complete in Phase 1)

```python
class User:
    def __init__(self, username, password):
        self.__username = username   # private
        self.__password = password   # private

    def get_username(self): return self.__username
    def get_password(self): return self.__password
    def set_username(self, username): self.__username = username
    def set_password(self, password): self.__password = password

    def authenticate(self, input_username, input_password):
        # case-sensitive, stripped comparison
        return (self.__username.strip() == input_username.strip() and
                self.__password.strip() == input_password.strip())

    def __str__(self): return f"User: {self.__username}"
```

## 6. Algorithm Design

### 6.1 Overall Control Flow

```algorithm
1. HoweAdmin.py starts
2. sys.path.insert(0, os.path.dirname(os.path.abspath(__file__))) — ensures imports resolve
3. Call login_control(ADMIN_CSV_PATH, MAX_LOGIN_ATTEMPTS, Admin)
4. If login succeeds → store admin_obj → show placeholder menu
5. If login fails (5 attempts exhausted) → print lockout message → exit
```

---

### 6.2 `read_csv(filepath)` Algorithm

```algorithm
Open file at filepath using csv.reader
Skip the first row (header)
Read remaining rows into a list of lists
Return the list
```

---

### 6.3 `login_control(filepath, max_attempts=5, user_class=None)` Algorithm

```algorithm
Determine column indices:
  If filepath == STUDENTS_CSV_PATH: username_idx=2, password_idx=3
  Else (admins.csv): username_idx=0, password_idx=1

Load credential_data = read_csv(filepath)
attempts = 0

While attempts < max_attempts:
    Prompt for username (strip)
    Prompt for password (strip)

    For each row in credential_data:
        stored = user_class(row[username_idx], row[password_idx])
        If stored.authenticate(input_username, input_password):
            Return (True, stored)

    attempts += 1
    remaining = max_attempts - attempts
    If remaining > 0: print "X attempt(s) remaining"
    Else: print "Account locked."

Return (False, None)
```

---

### 6.4 Main Executable Algorithm (HoweAdmin.py)

```algorithm
Display program header
Print "Please log in to continue."

(success, admin_obj) = login_control(ADMIN_CSV_PATH, MAX_LOGIN_ATTEMPTS, Admin)

If not success:
    Print "Access denied. Exiting."
    sys.exit()

Print "Welcome, {admin_obj.get_username()}!"
Print "Admin menu coming in Phase 2..."
```

## 9. Edge Cases & Validation

| Scenario | Handling |
| --- | --- |
| Admin enters empty username or password | Treated as invalid credentials; counts as a failed attempt |
| `admins.csv` file not found | `read_csv()` raises FileNotFoundError — handle with clear error message |
| Username matches but password does not | Count as failed attempt; do not reveal which field was wrong |
| Admin enters correct credentials on attempt 5 | Login succeeds; do not lock out |
| Admin exhausts all 5 attempts | Print lockout message and exit |
| Extra whitespace in input | `.strip()` on all inputs (handled in `authenticate()`) |
| Case sensitivity | Usernames and passwords are case-sensitive (exact match) |

## 12. Conceptual Summary

Phase 1 establishes both the security layer and the OOP architecture. The `User` class encapsulates credential storage and authentication. `Admin` and `Student` extend it with domain-specific behavior that grows across phases. All three classes are defined now so the inheritance hierarchy is complete from the start. The `login_control()` utility function accepts a `user_class` parameter so it works for both admin and student login without modification — the same function is reused in Phase 3 with `Student` passed as the class. All constants live in `HoweUtil.py` as the single source of truth for both executables.
